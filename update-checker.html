GUIDE

How to Update Your Skill Files — Read This First

↓

Your skill files are living documents. As your thinking evolves, your methods sharpen, and your experience deepens — the files should reflect that. But updates carry risk. The execution layer especially changes frequently, and a new tactic or framework added in isolation can quietly contradict a belief you committed to in your worldview layer.

WhenUpdate when your thinking has genuinely changed — not just your tactics
A new tool, a new technique, a new framework you've picked up belongs in execution.md only if it's consistent with your existing philosophy. If it requires you to change how you see the problem or the people you serve, update worldview.md first. Execution never leads. Philosophy leads.
DoUpdate in the right layer
worldview.md → shifts in what you believe. brand.md → changes in how you show up or communicate. execution.md → new methods, tools, frameworks, or process refinements. system.md → new task types or routing adjustments. Every update belongs in exactly one file.
Don'tNever add without running this checker first
Every proposed addition to any skill file should pass through this decision tree before being added. worldview.md is always the authority. If something contradicts it, either don't add it — or update worldview.md first and reconcile the change intentionally.
DoTranslate borrowed frameworks into your own words before adding
If you can't explain why an addition belongs in your system in one sentence, it probably doesn't — yet. Write updates in the same register as the rest of your file. If you're using jargon from a course or a book, translate it first.
